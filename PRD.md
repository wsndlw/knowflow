# knowflow 产品需求文档（PRD）

## 一、项目背景

### 1.1 问题与机会

企业/机构内部积累了大量业务文档、经验知识和对话记录，但存在以下问题：

- **知识分散**：文档散落在各处，员工难以快速找到需要的信息
- **重复劳动**：相同问题被反复询问，缺乏统一的知识沉淀机制
- **质量参差**：知识更新不及时，答案准确性无法保证
- **权限混乱**：敏感信息可能被无关人员访问

### 1.2 解决方案

构建企业级 AI 知识库管理平台，实现：

- 集中管理各类业务文档与知识条目
- 基于 RAG 的智能问答，带引用来源与可信度评估
- 知识自动提炼与持续改进闭环
- 严格的三层权限控制（超管/部门管理员/普通用户）
- 知识库级别的可见范围隔离（公开/部门/受限）

### 1.3 目标用户

- **超级管理员**：负责平台整体管理、部门划分、用户管理、模型配置
- **部门管理员**：负责本部门知识库的创建、内容管理、成员授权
- **普通用户**：使用知识库进行问答、浏览文档、反馈问题

## 二、核心功能需求

### 2.1 账号与权限管理

**用户角色**

- **超级管理员（super_admin）**：平台唯一最高权限，管理所有部门、用户、知识库、模型配置
- **部门管理员（department_admin）**：管理本部门用户、本部门知识库、部门级统计
- **普通用户（user）**：使用知识库、创建个人 Agent、参与对话

**知识库可见范围**

- **公开（public）**：全员可访问，适用于公司通用知识（如员工手册、制度文档）
- **部门（department）**：仅本部门成员可访问，适用于部门专属知识（如技术文档、业务流程）
- **受限（restricted）**：仅手动添加的成员可访问，适用于敏感或项目级知识（如商业秘密、项目资料）

**权限校验原则**

- 所有权限判断由后端兜底，前端仅控制 UI 显示
- RAG 检索必须在权限过滤后进行，不能把无权限文档送入上下文
- 管理后台操作记录完整审计日志

---

### 2.2 知识库管理

**知识库创建与编辑**

- 超管/部门管理员可创建知识库，设置名称、描述、可见范围、封面
- 支持为知识库添加标签（如"技术文档"、"产品手册"），便于分类筛选
- 支持软删除：删除的知识库进入回收站，可恢复或永久删除

**知识库状态**

- **草稿（draft）**：创建中，未发布
- **已发布（published）**：正式可用
- **已归档（archived）**：不再活跃，只读不可编辑

**知识库成员管理**

- 受限知识库支持手动添加成员（用户级授权）
- 成员列表展示姓名、邮箱、加入时间
- 支持批量添加、移除成员

---

### 2.3 文档上传与处理

**支持的文档格式**

- **文本类**：TXT、Markdown
- **办公文档**：PDF、DOCX
- **结构化数据**：CSV、Excel
- **图片**：PNG、JPG（需配置 OCR 模型）

**文档处理链路**

1. **上传入队**：用户上传文档后进入异步处理队列（BullMQ + Redis）
2. **解析提取**：根据文件类型调用对应解析器（pdf-parse、mammoth、xlsx 等）
3. **OCR 增强**：
   - 整图上传直接调用 OCR
   - 扫描件 PDF（文本层稀疏）逐页渲染后 OCR
   - PDF/DOCX 内嵌图片提取后 OCR
4. **父子分段**：
   - 按标题层级切节
   - 语义打包成父块（目标 2600 字，上限 4000 字）
   - 父块切子块（目标 900 字 + 120 字重叠）
5. **批量向量化**：子块嵌入为 1024 维向量，存入 pgvector
6. **进度实时推送**：通过 SSE 将处理进度推送到前端
7. **触发知识提炼**：文档处理完成后自动生成知识提炼候选

**文档状态**

- **处理中（processing）**：解析、分段、向量化进行中
- **已完成（completed）**：可用于检索
- **失败（failed）**：处理异常，展示错误信息

---

### 2.4 知识条目管理

**知识条目（Knowledge Item）**

知识条目是人工维护的结构化知识卡片，适用于 FAQ、政策解读、概念定义等场景。

**字段**

- 标题、内容、标签、来源说明
- 可能的问题（用户可能这样问）
- 创建者、审核者、发布时间

**状态流转**

```
草稿（draft）→ 待审核（pending_review）→ 已发布（published）→ 已下架（unpublished）/ 已归档（archived）
```

**知识条目向量化**

- 发布时自动嵌入为 1024 维向量
- 与文档分段一起参与检索召回

---

### 2.5 RAG 智能问答

**检索策略：三路召回**

1. **向量检索**：语义相似度匹配，Top 20
2. **全文检索（FTS）**：关键词精确匹配，Top 20
3. **知识条目检索**：结构化知识卡片，Top 10

**检索优化**

- **合并去重**：三路结果按 `parentChunkId` 或 `knowledgeItemId` 去重
- **真实 Rerank**：调用 DashScope Rerank API，取前 30 条重排序后保留 Top 10
- **父子扩展**：子块命中后扩展回父块全文，保证上下文完整
- **Token 预算**：上下文总 Token 不超过 6000，超出时按分数裁剪

**对话记忆：双层架构**

- **短期窗口**：最近 6 条原文消息，保证上下文连贯
- **滚动摘要**：超过 10 条消息后，Worker 异步生成摘要，压缩早期对话
- 摘要完全异步，不阻塞回答流程

**流式回答**

- 基于 LangGraph 固定图（12 节点）执行问答流程
- 每个节点进出发送 SSE 事件，前端实时展示进度
- 答案流式返回，逐 token 展示

**引用来源与可信度**

- 每个答案标注引用来源（文档/知识条目），带 `[n]` 引用标号
- 根据证据强度评估可信度：强/中/弱/无答案
- 无答案兜底：检索为空、分数过低、LLM 拒答时返回统一提示

**反馈与改进**

- 用户可对答案点赞/点踩
- 点踩后可选填纠错内容
- 点踩/纠错自动触发知识提炼候选生成

---

### 2.6 专家 Agent

**Agent 类型**

- **全局 AI 助手**：检索用户有权限的所有知识库，适用于跨库问答
- **知识库官方 Agent**：绑定单个知识库，由管理员创建和管理
- **个人 Agent**：用户自己创建，仅自己可见，可自定义提示词

**一键生成 Agent**

- 管理员在知识库内点击"一键生成 Agent"
- AI 自动生成：名称、描述、系统提示、开场白、推荐问题
- 管理员审核后发布，或手动编辑后发布

**Agent 配置**

- 名称、描述、系统提示词
- 开场白、推荐问题列表
- 答案风格（专业/友好/简洁）
- 是否允许上传附件
- 模型供应商与模型名称（可选，默认继承全局配置）

**Agent 状态**

- **草稿（draft）**：编辑中，仅创建者可见
- **已发布（published）**：对知识库成员可见

---

### 2.7 知识自动提炼

**触发信号（四路）**

1. **文档完成**：新文档处理完成后自动生成候选知识
2. **无答案缺口**：用户提问后检索为空或可信度过低
3. **点踩反馈**：用户对答案不满意
4. **主动纠错**：用户提交纠错内容

**候选生成**

- AI 根据来源（文档内容/问答记录/纠错）生成候选知识条目
- 状态：`pending`（待生成）→ `processing`（生成中）→ `candidate_ready`（待审核）/ `failed`（生成失败）
- 生成的内容：标题、内容、标签、可能的问题

**人工审核**

- 管理员在"知识改进"页面查看候选
- 可审核通过（发布到正式知识库）或拒绝
- **AI 只生成候选，不直接写入正式库**

**质量验证**

- 非文档来源的知识条目发布后，7 天后自动验证是否仍有类似问题答不上
- 标记为 `verified`（有效）或 `still_failing`（仍失败）

---

### 2.8 知识关系思维导图

**AI 自动生成**

- 管理员点击"生成思维导图"
- AI 根据知识库内的文档和知识条目生成主题节点
- 自动关联文档和知识条目到对应主题

**手动编辑**

- 支持添加/删除/重命名主题节点
- 支持调整节点层级关系
- 支持手动关联/取消关联文档和知识条目

**可视化展示**

- 树状结构展示知识库的知识结构
- 点击节点查看关联的文档/知识条目列表
- 支持展开/折叠子节点

---

### 2.9 统计与治理

**使用热度可视化**

- 文档使用趋势：查看次数、引用次数、时间分布
- 知识条目使用趋势：被检索次数、被引用次数
- Agent 使用趋势：对话次数、用户活跃度

**检索测试**

- 管理员输入测试问题，查看检索结果
- 展示三路召回结果、Rerank 分数、父子扩展效果
- 用于调试检索质量，不调用 LLM 生成答案

**操作审计日志**

- 记录所有敏感操作：创建/删除知识库、上传/删除文档、发布知识条目、修改权限
- 字段：操作人、操作类型、目标对象、结果、IP、User-Agent、时间
- 支持按时间、操作人、操作类型、结果筛选

**知识库回收站**

- 删除的知识库进入回收站（软删除）
- 超管/部门管理员可在回收站恢复或永久删除
- 永久删除需二次确认，且级联删除所有文档、知识条目、对话记录

---

### 2.10 管理后台

**部门管理**

- 超管创建/编辑/删除部门
- 为部门分配部门管理员
- 查看部门下的用户、知识库

**用户管理**

- 超管/部门管理员创建/编辑用户
- 设置用户角色（超管/部门管理员/普通用户）
- 绑定用户所属部门
- 停用/启用用户账号

**模型配置**

- 按用途（对话/Embedding/Rerank/知识生产/OCR）配置模型
- 支持多供应商（阿里云百炼、OpenAI-compatible 中转站）
- 支持热切换：修改配置后无需重启服务
- API Key 加密存储，前端脱敏展示

**全局统计**

- 平台级别的知识库数量、文档数量、对话次数
- 模型调用次数、Token 消耗统计
- 用户活跃度分布

## 三、非功能需求

### 3.1 安全性

- 所有 API 接口必须验证用户身份（JWT Token）
- 权限判断在后端兜底，不信任前端
- RAG 检索必须先过滤权限，不能泄露无权限文档
- 敏感字段（API Key、密码）加密存储
- 操作日志记录完整审计信息

### 3.2 性能

- 文档处理异步化，不阻塞用户上传
- 对话摘要异步生成，不阻塞回答
- 向量检索响应时间 < 500ms（90 分位）
- 流式回答首 Token 延迟 < 2s

### 3.3 可扩展性

- 模型供应商可插拔，支持运行时切换
- 向量空间统一（1024 维），便于模型迁移
- 文档处理器支持扩展新格式

### 3.4 可观测性

- 每次问答记录完整 trace（检索上下文、prompt 快照、模型配置、耗时）
- SSE 推送各节点执行状态，前端实时展示
- 错误统一处理，返回友好提示

## 五、技术约束

### 5.1 技术栈

- **前端**：Next.js（App Router）、React、shadcn/ui、Radix UI、Tailwind CSS
- **后端**：NestJS、LangGraph.js（Agent 运行时）
- **异步任务**：BullMQ + Redis
- **数据库**：PostgreSQL + pgvector（向量检索）+ Full Text Search（全文检索）
- **ORM**：Drizzle ORM
- **模型**：阿里云百炼 DashScope（对话/Embedding/Rerank/知识生产/OCR）

### 5.2 架构原则

- Monorepo 布局（pnpm workspace）
- 全 TypeScript 起步，前后端同语言
- 前后端通过 `packages/shared` 共享类型与 Zod schema
- 数据库 schema 与迁移由 `packages/db` 统一管理
- 异步任务与后端共享 Redis、PostgreSQL

## 六、用户场景示例

### 6.1 场景一：新员工入职培训

**角色**：普通用户（新员工）

**流程**：

1. 登录系统后，看到"员工手册"知识库（公开知识库）
2. 点击进入知识库，看到官方 Agent"HR 小助手"
3. 提问："试用期多久？"
4. 系统检索员工手册，返回答案并标注来源文档
5. 新员工点赞，表示满意

### 6.2 场景二：技术文档管理

**角色**：部门管理员（技术部门）

**流程**：

1. 创建知识库"技术文档"，可见范围设为"部门"
2. 上传 20 份 PDF 技术文档，系统异步处理
3. 处理完成后，点击"一键生成 Agent"，AI 生成"技术助手"
4. 审核后发布，部门成员可使用"技术助手"提问
5. 查看统计页面，发现某份文档使用率最高

### 6.3 场景三：知识质量改进

**角色**：知识库管理员

**流程**：

1. 收到通知：用户对某个答案点踩
2. 进入"知识改进"页面，查看 AI 生成的候选知识
3. 审核候选内容，发现确实有用，点击"发布"
4. 该知识条目立即可用于检索
5. 7 天后系统验证，发现类似问题答对了，标记为"已验证"

## 八、附录

### 8.1 术语表

- **知识库（Knowledge Base）**：知识的容器，包含文档、知识条目、专家 Agent
- **文档（Document）**：用户上传的文件，经解析、分段、向量化后可用于检索
- **知识条目（Knowledge Item）**：人工维护的结构化知识卡片
- **专家 Agent（Expert Agent）**：绑定知识库的对话入口，负责检索与回答
- **父子分段（Parent-Child Chunking）**：子块用于精确召回，父块用于完整上下文
- **三路召回（Triple Retrieval）**：向量检索 + 全文检索 + 知识条目检索
- **Rerank**：对初排结果重新排序，提高相关性
- **RAG（Retrieval-Augmented Generation）**：检索增强生成，基于检索上下文生成答案
- **可信度（Confidence）**：根据证据强度评估答案质量（强/中/弱/无答案）
